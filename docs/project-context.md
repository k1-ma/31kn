# Контекст проекта «Koshyk» — пояснение для написания дипломной работы

> Это не инструкция «как писать диплом». Это **рабочий контекст** со стороны программиста:
> что за проект, какой стек использовался и **почему именно такой**, как всё устроено внутри,
> что переделывалось и какие технические нюансы важны. Цель — чтобы человек, который пишет
> по проекту дипломную, понимал суть каждого решения и мог грамотно его описать и защитить.

---

## 1. Что это за проект

**Koshyk** — веб-приложение для учёта личных финансов: гаманці (кошельки), категории,
транзакции, бюджеты, цели (goals), повторяющиеся операции (recurring), долги (debts).
Mobile-first PWA (устанавливается как приложение, работает офлайн). Интерфейс двуязычный:
украинский (по умолчанию) и английский.

Архитектурно это классическое **клиент-серверное SPA**: React-фронтенд + REST API на
Express + база PostgreSQL. Принцип — **server-authoritative**: сервер является единственным
источником истины, клиент — тонкий кеш поверх REST.

**Главный технический сюжет, который стоит понимать:** проект вырос из другого приложения —
трейдерского журнала **TradeCRM**. У того была «наивная» архитектура хранения (всё состояние
пользователя — один большой JSON-объект, **v1**), которая плохо масштабировалась. Проект
переименовали и переписали в финансовый трекер Koshyk, а архитектуру данных перевели на
**нормализованную модель по сущностям с TanStack Query (v2)**. Понимание перехода v1 → v2 —
ключ к пониманию всего проекта (см. раздел 3).

## 2. Стек технологий и почему выбрано именно это

### Frontend
- **React 18.3** — компонентный UI. Выбран как индустриальный стандарт для SPA: большая
  экосистема, хуки, декларативный рендеринг. Используется ленивая загрузка страниц
  (`lazy()` + `Suspense`) — чтобы первый экран грузился быстро, а тяжёлые страницы
  подтягивались по требованию.
- **Vite 5.3** (сборщик) + плагин React — быстрый дев-сервер с горячей заменой модулей (HMR),
  быстрая прод-сборка. В дев-режиме проксирует `/api` на Express (порт 8080). В проде:
  ручное разбиение бандла на чанки (`recharts`, `framer-motion`, `lucide`, `router`, `react`
  выносятся отдельно, чтобы тяжёлые библиотеки не блокировали ленивую загрузку страниц),
  вырезание `console`/`debugger`, отключённые sourcemap (меньше вес, не утекает исходник).
- **React Router 6.26** — клиентская маршрутизация. Разделение зон: публичная (лендинг,
  логин, регистрация), пользовательская (`/app/*`, защищена `RequireAuth`) и админская
  (`/admincrm-panel/*`, защищена `RequireAdmin`).
- **@tanstack/react-query 5.101** — **ключевая библиотека**. Управляет кешем серверных
  данных, делает оптимистичные обновления и инвалидацию. Именно она в v2 заменила
  самодельный слой синхронизации (см. раздел 3). Почему её: она снимает с разработчика
  всю рутину кеширования/перезапросов/состояния загрузки и даёт оптимистичный UI «из коробки».
- **Tailwind CSS 3.4** (+ PostCSS, autoprefixer) — utility-first стили. Почему: быстро
  верстать, единые токены дизайна, тёмная тема по классу, mobile-first брейкпоинты.
- **framer-motion 11** — анимации (выезжающая снизу форма-«шторка» BottomSheet с перетягиванием
  для закрытия, выпадающее меню мобильной навигации, переходы страниц). Уважает системную
  настройку `prefers-reduced-motion`.
- **recharts 3.7** — графики (столбчатый график денежного потока по месяцам, круговой по
  категориям расходов; в админке — графики использования). Почему: декларативный API поверх
  D3, хорошо ложится на React.
- **lucide-react** — набор SVG-иконок (чистые, лёгкие, tree-shakeable).
- **Шрифты:** Inter (основной текст), Geist / Geist Mono (заголовки и моноширинные
  «табличные» цифры — чтобы суммы ровно выравнивались в столбик). Раньше для заголовков был
  Space Grotesk, в ходе редизайна заменён на Geist.
- **vite-plugin-pwa (Workbox)** — service worker, манифест, офлайн-режим, авто-обновление.

### Backend
- **Node.js ≥18 + Express 4.19** — HTTP-сервер. Express выбран как самый распространённый
  минималистичный фреймворк; приложение собирается фабрикой `createApp()` с явным порядком
  middleware.
- **PostgreSQL + pg 8.13** (node-postgres) — реляционная СУБД. Выбрана из-за надёжности,
  индексов, транзакций и того, что нормализованная модель (v2) идеально на неё ложится.
  Пул соединений (по умолчанию 15), SSL для не-localhost, ограничение времени запроса
  (statement_timeout 10с).
- **bcryptjs 2.4** — хеширование паролей (cost factor 14). Нюанс: старые хеши (cost 10–13)
  при успешном входе незаметно перехешируются на cost 14 — постепенный апгрейд безопасности
  без принудительного сброса паролей.
- **helmet 8.1** — security-заголовки (Content-Security-Policy и т.д.).
- **cors 2.8** — CORS с белым списком разрешённых источников; запросы того же источника
  (same-origin) разрешены всегда.
- **otplib 12 + qrcode 1.5** — двухфакторная аутентификация по TOTP (как Google
  Authenticator) и генерация QR-кода для её настройки.
- **resend 4.0** — отправка писем (подтверждение e-mail, сброс пароля, смена e-mail). Если
  ключ не задан — письма просто не отправляются, приложение работает без них.
- **dotenv** — переменные окружения.

### Тестирование и инструменты
- **Vitest 4.1** (+ jsdom) — модульные тесты (быстрый, нативно понимает ESM, тот же движок
  что и Vite).
- **ESLint 9** (flat config) — статический анализ; настроен так, чтобы ловить реальные ошибки
  и **запрещать импорт старых антипаттерн-модулей** (см. раздел 7).
- **nodemon, concurrently** — для разработки (авто-перезапуск сервера + параллельный запуск
  Vite и Express).

### Деплой
- **Vercel** — API как одна serverless-функция (`api/index.js`), статика из `dist/`,
  маршрутизация в `vercel.json`, edge-middleware (`middleware.js`) для редиректа на
  канонический домен. База Postgres — внешняя (Neon / Supabase / RDS / локальная).

## 3. Самое важное: переход от v1 (антипаттерн) к v2 (правильная архитектура)

Это сердце проекта. В дипломе это лучший материал для раздела «проектирование» и «выводы».

### Как было (v1, наследие TradeCRM)
Всё состояние пользователя — **один большой JSON-объект** (`states.state_json`, тип JSONB в
Postgres). Все кошельки, все транзакции, бюджеты и т.д. лежали в одной строке таблицы.
Клиент хранил копию этого блоба в браузере (IndexedDB) и синхронизировал с сервером
самодельным механизмом: стратегия «last-write-wins», «чанковая» досылка для блобов,
которые не влезали в лимит размера запроса, и клиентская логика слияния конфликтов
(тумбстоуны, счётчики версий, обработка ошибок 409).

### Почему это плохо (мотивация переделки — обязательно описать)
1. **Неограниченный рост строки.** Блоб одного пользователя — это весь его аккаунт. У
   активных пользователей он упирался в лимит размера тела HTTP-запроса, и приходилось
   «нарезать» его на части просто чтобы сохранить.
2. **Запись всего ради изменения одного.** Поменял одну транзакцию — переписывается весь
   блоб целиком. Стоимость операции = O(размер состояния), а не O(изменение).
3. **Гонки и потеря данных.** Два устройства редактируют одновременно → затирают правки друг
   друга. Чтобы это лечить, клиент оброс сложной логикой слияния и версионирования.
4. **Аналитика тормозила.** Чтобы в админке посчитать число транзакций пользователя, нужно
   было сканировать мегабайтный JSON (`jsonb_array_elements`) — запросы упирались в таймаут.

### Как стало (v2, текущая модель)
- **Каждая коллекция — своя нормализованная таблица** в Postgres с внешним ключом на `users`
  и нужными индексами. Единого «блоба» больше нет.
- **Клиент — тонкий кеш поверх REST.** TanStack Query держит данные; каждая мутация — это
  один REST-запрос на изменение одной строки, с оптимистичным обновлением интерфейса.
  Самодельной синхронизации и IndexedDB-блоба больше нет.
- **Настройки** (базовая валюта, тема) — единственное «не-коллекционное» состояние, лежит в
  таблице `user_preferences` (одна строка на пользователя, маленький JSONB).

### Что это дало (улучшения — для выводов)
- Рост данных ограничен и индексирован; огромные аккаунты больше не ломают сохранение.
- Изменение стоит **одну строку** (O(изменение)).
- Параллельные правки затрагивают **разные строки** → гонок нет, сложную логику слияния
  удалось просто удалить.
- Аналитика админки — обычные индексированные `COUNT(*)`, без сканов JSON.

### Нюанс миграции
Старая таблица `states` оставлена как **исторический артефакт** (в работающем коде её никто
не читает). Скрипт `server/scripts/backfill-entities.js` идемпотентно переносит старые блобы
в новые таблицы — то есть его можно запускать повторно без дублей.

## 4. Модель данных (что лежит в базе)

**Все деньги — целые числа в копейках/центах** (`BIGINT`: `amount_cents`, `balance_cents`,
`limit_cents`, `target_cents`). Это принципиально: дробные суммы (float) дают ошибки
округления, поэтому деньги хранят и считают в целых центах, а в человеческий вид переводят
только на границе интерфейса (`src/lib/money.js`). При чтении из API центы приводятся обратно
к числу (`Number`), чтобы на клиенте `+` складывал числа, а не склеивал строки.

У всех финансовых таблиц есть: `id` (текстовый, **генерируется клиентом** — это нужно для
оптимистичного UI, чтобы не ждать ответа сервера), `user_id`, `created_at`, `updated_at` и
`deleted_at` (мягкое удаление → «Корзина»; отдельная операция `purge` удаляет окончательно).

| Коллекция | Таблица | Ключевые поля |
|---|---|---|
| Кошельки | `wallets` | `balance_cents`, `currency`, `type`, `sort_order`, `is_archived` |
| Категории | `categories` | `kind` (доход/расход), `parent_id` (вложенность), `sort_order` |
| Транзакции | `transactions` | `type` (доход/расход/перевод), `amount_cents`, `wallet_id`, `to_wallet_id`, `category_id`, `date`, `tags` |
| Бюджеты | `budgets` | `category_ids` (список id), `period`, `limit_cents`, `alert_at` (% порог) |
| Цели | `goals` | `target_cents`, `current_cents`, `target_date` |
| Повторяющиеся | `recurring_rules` | `template`, `frequency`, `every`, `next_run_at` |
| Долги | `debts` | `direction` (я должен / мне должны), `amount_cents`, `due_date`, `is_settled` |

Плюс служебные таблицы: `users`, `sessions`, токены 2FA и резервные коды, сброс пароля,
`notifications`, `admin_logs` (аудит), `ip_bans`, `usage_daily` (метрики), `idempotency_keys`,
`user_preferences`, `exchange_rates_cache`.

**Важный нюанс про JSONB.** JSONB в проекте не запрещён полностью — запрещено хранить в нём
**коллекции сущностей** (это и был антипаттерн v1). Разрешён JSONB только для мелких
«сопутствующих» данных одной строки: настройки пользователя, теги транзакции, список id
категорий в бюджете, шаблон повторяющегося правила, данные уведомления.

## 5. REST API и универсальная CRUD-фабрика (интересное инженерное решение)

Вместо того чтобы вручную писать одинаковые CRUD-роуты для каждой из 7 сущностей, в проекте
сделана **обобщённая фабрика**: `server/lib/entityRouter.js` строит набор маршрутов из
описания сущности `server/lib/entityConfig.js`. Каждое поле описывается тройкой
`{ key, col, kind }` (имя в API, колонка в БД, тип). Это автоматизирует:
- маппинг строки БД → объект для клиента (центы → число, даты → строки),
- валидацию (обязательные поля, диапазоны сумм),
- генерацию одинаковых эндпоинтов.

Добавить новую сущность = добавить таблицу + одну запись в конфиг + запрос на клиенте.
Отдельный роут-файл писать не нужно. Это хороший пример **принципа DRY** и метапрограммирования
для диплома.

Эндпоинты на каждую сущность:
```
GET    /api/:entity            список (включая удалённые — для Корзины)
POST   /api/:entity            создание (id от клиента; повторный POST идемпотентен)
PUT    /api/:entity/:id        обновление
DELETE /api/:entity/:id        мягкое удаление
POST   /api/:entity/:id/restore   восстановление из Корзины
DELETE /api/:entity/:id/purge     окончательное удаление
GET/PUT /api/preferences          настройки
POST   /api/import                восстановление всего аккаунта из бэкапа (транзакционно)
```
Важно: `user_id` всегда берётся **из сессии**, никогда из тела запроса — чтобы один
пользователь не мог подделать данные другого. Мутации несут заголовок `Idempotency-Key`
(защита от дублей при повторной отправке) и ограничены по частоте (60 записей/мин).
`POST /api/import` — единственное место, где легитимно пишется весь аккаунт сразу
(восстановление из бэкапа), и делается это в транзакции.

## 6. Как устроены ключевые подсистемы

### Безопасность и аутентификация
- Пароли — bcrypt (cost 14). При входе несуществующего пользователя всё равно выполняется
  фиктивное сравнение хеша — **защита от перечисления пользователей по времени ответа**
  (timing-safe).
- Сессии — подписанная cookie (HMAC), хранение в таблице `sessions`, срок 1 день или 30 дней
  («запомнить меня»).
- **Двухфакторная аутентификация (TOTP):** секрет шифруется (AES-256-GCM), есть 10 резервных
  кодов, защита от повторного использования одного и того же кода, вход через временный
  «тикет-челлендж» (10 минут).
- Верификация e-mail, сброс пароля, смена e-mail с подтверждением; кулдауны на смену
  отображаемого имени (7 дней) и логина (30 дней).
- **Rate limiting** (ограничение частоты): логин 10/5мин на IP, 2FA 5/15мин, регистрация
  5/час, запись данных 60/мин на пользователя.
- **Идемпотентность:** повторный запрос с тем же ключом возвращает закешированный ответ
  (через advisory-lock Postgres, чтобы параллельные повторы не создавали дубли). Это важно
  для надёжности при плохой сети и для оптимистичного UI.

### Клиент
- Точка входа → роутер → оболочка `KoshykApp` (провайдеры тостов и данных, PIN-замок,
  навигация: боковая на десктопе, нижняя на мобильном).
- Слой данных: `src/queries/finance.js` (по одному запросу на коллекцию) и фасад
  `FinanceProvider` (`src/lib/finance/store.jsx`) с методами
  `upsert / remove / restore / purge / setPrefs / importBackup`. Нюанс: при переходе на v2
  **публичный API фасада сохранили прежним** — страницы не переписывали, поменяли только
  «внутренности» (REST вместо блоба). Это пример того, как рефакторинг архитектуры можно
  провести без переписывания всего UI.
- Страницы: Дашборд (нетто-капитал по валютам с пересчётом по курсу, онбординг для новичка),
  Транзакции (+ форма-шторка с цифровой клавиатурой), Кошельки, Категории, Бюджеты (кольца
  прогресса, алерты при 80%/100%), Цели, Повторяющиеся, Долги, Аналитика (графики), Настройки
  (тема, валюта, язык, CSV-импорт, JSON-бэкап, PIN, 2FA), Корзина, Уведомления.
- Деньги: `src/lib/money.js` (`toCents`, `formatMoney` через `Intl.NumberFormat` под локаль).
- Мультивалютность: курсы тянутся с публичного API (exchangerate.host), кешируются в браузере;
  при сбое API приложение продолжает работать (мягкая деградация).
- **Финансовая логика вынесена в чистые функции** (`src/lib/finance/`: расчёт баланса
  кошелька, прогресса бюджета, материализация повторяющихся операций, парсинг CSV, бэкап,
  алерты). Это удобно тестировать — именно эти модули покрыты юнит-тестами.

### PWA / офлайн
- Service worker (Workbox): авто-обновление, навигационный fallback на `index.html`.
  Стратегии кеша: аутентификация и мутации — только сеть (NetworkOnly), GET-запросы данных —
  «сначала сеть» с таймаутом, шрифты и картинки — «сначала кеш». Тема применяется
  preload-скриптом ещё до загрузки React (чтобы не мигал белый экран на тёмной теме).

### Уведомления
- Серверная таблица уведомлений (типы: предупреждение/превышение бюджета, наступление
  повторяющейся операции, достижение цели, напоминание, системное сообщение) с дедупликацией,
  пагинацией, пометкой «прочитано».

## 7. Качество и «защита от регресса» (governance)

Это сильная часть проекта и хорошая тема для раздела «обеспечение качества».

- **Тесты (Vitest + jsdom):** ~12 файлов, ~90+ кейсов. Покрыто самое критичное и
  «математическое»: денежная арифметика, курсы валют, расчёты бюджетов/балансов,
  повторяющиеся операции, CSV-импорт, долги, бэкап, и серверный маппинг сущностей.
- **CI/CD** (GitHub Actions): на каждый push/PR последовательно — проверка антипаттернов →
  ESLint → тесты → сборка.
- **Структурный «guard»** (`scripts/check-patterns.sh`): grep-скрипт, который **роняет
  сборку**, если кто-то пытается вернуть старые антипаттерны v1 — вызовы удалённых
  блоб-эндпоинтов, самодельные функции синхронизации/слияния, обращение к удалённому
  IndexedDB-хранилищу, запись целой коллекции одним куском. Это «исполняемая документация»:
  правила архитектуры не на словах, а проверяются машиной.
- **ESLint** дополнительно запрещает (как ошибку) импорт старых модулей синхронизации.
- **Правила проекта `CLAUDE.md`** — 5 «жёстких» правил: (1) не хранить коллекции в JSONB-блобе;
  (2) деньги — целые центы; (3) сервер — источник истины, синхронизация только через
  TanStack Query; (4) изменение стоит O(изменение); (5) ошибки структурированные, со стабильным
  кодом, без логирования секретов.

Смысл всего этого: проект уже один раз болезненно ушёл от плохой архитектуры, и эти проверки
**не дают ей вернуться**.

## 8. Как проект развивался (хронология)

1. **TradeCRM** — исходный трейдерский журнал на блоб-модели v1 (с самодельной
   синхронизацией; её последние «костыли» в истории git и объясняют, зачем нужен был ребилд).
2. **Ребрендинг TradeCRM → Koshyk** — полная смена предметной области на личные финансы,
   mobile-first интерфейс, форма быстрого ввода транзакций, двуязычие (uk/en).
3. **Раунды функционала** — повторяющиеся операции, CSV-импорт, мультивалютный нетто-капитал,
   JSON-бэкап, тосты, алерты бюджетов, PIN-замок, фильтры/теги/сортировка, цели, уведомления,
   долги, скелетоны загрузки.
4. **Редизайн «Calm Fintech v2»** — индиго-бренд, градиенты, шрифт Geist, кольца прогресса.
5. **Миграция v1 → v2** — нормализованные таблицы + универсальный CRUD на сервере и TanStack
   Query вместо блоб-синхронизации на клиенте. Внешне UI не изменился, внутри — другая
   архитектура.
6. **Governance** — правила, ESLint, guard-скрипт, CI, чтобы v2 не «сползла» обратно в v1.
7. **Полировка** — онбординг, скелетоны, инлайн-валидация форм, тема на весь апп, починка
   админ-дашборда и графиков.

## 9. Технические нюансы, которые легко упустить (но они важны для понимания)

- **Клиентские id и оптимистичный UI.** id создаёт клиент, поэтому интерфейс обновляется
  мгновенно, не дожидаясь сервера; POST идемпотентен (`ON CONFLICT`), так что повтор запроса
  не создаёт дубль.
- **Мягкое удаление vs «purge».** Удаление — это не `DELETE`, а проставление `deleted_at`
  (запись уходит в Корзину и её можно восстановить). Реально из БД строку убирает только
  `purge`. Двухшаговое удаление — защита от случайной потери данных.
- **Почему деньги в центах.** Чтобы исключить ошибки округления float. Это правило проходит
  сквозь весь стек: БД (BIGINT) → API (Number) → клиент (`money.js`).
- **Почему сохранили публичный API store при миграции.** Чтобы переписать архитектуру данных,
  не трогая десятки страниц UI — поменяли только реализацию фасада.
- **Мягкая деградация.** Нет ключа почты → письма не шлются, но приложение работает. Упал API
  курсов → суммы показываются без пересчёта. БД недоступна → сервер отвечает структурированной
  ошибкой 503, клиент это понимает.
- **Легаси-артефакты переходного периода.** В коде остались отдельные следы прошлого
  (упоминания «чанковой» синхронизации в комментариях PWA-конфига, заготовки Google OAuth,
  отдельные лимитеры). При описании проекта их стоит трактовать как остатки переходного
  периода — **действующей является архитектура v2** (см. также `docs/ARCHITECTURE.md`).

## 10. Краткая «шпаргалка» (одной строкой по слоям)

| Слой | Технология | Зачем |
|---|---|---|
| UI | React 18 + React Router 6 | компонентный SPA, ленивые страницы |
| Стили | Tailwind + framer-motion | быстрая mobile-first вёрстка, анимации |
| Данные (клиент) | TanStack Query | кеш + оптимистичные мутации вместо самодельной синхронизации |
| Графики | recharts | аналитика расходов/доходов |
| Сборка/PWA | Vite + vite-plugin-pwa | быстрая сборка, офлайн, установка как приложение |
| API | Node + Express | REST, универсальная CRUD-фабрика |
| БД | PostgreSQL (pg) | нормализованные таблицы по сущностям |
| Безопасность | bcrypt, otplib, helmet, rate limit, idempotency | пароли, 2FA, заголовки, защита от злоупотреблений и дублей |
| Почта | Resend | подтверждения и сброс пароля |
| Качество | Vitest, ESLint, check-patterns.sh, CI | тесты + защита от возврата антипаттернов |
| Деплой | Vercel + внешний Postgres | serverless API + статика |

Главная мысль для защиты: **проект демонстрирует осознанный инженерный выбор** — переход от
наивной «всё в одном JSON» модели к нормализованной архитектуре, с измеримыми улучшениями
(стоимость записи, отсутствие гонок, скорость аналитики, масштабируемость) и с механизмами,
которые не дают этим улучшениям откатиться назад.
