# Architecture — Koshyk v2

Koshyk is a personal finance tracker (wallets, categories, transactions,
budgets, goals, recurring rules, debts). This document describes the **v2 data
architecture** introduced by the per-entity migration, and — importantly for
context — what it replaced.

## TL;DR

- **Server is the source of truth.** Each finance collection lives in its own
  normalized Postgres table with a foreign key to `users` and the right
  indexes. There is no single "state blob".
- **The client is a thin cache over REST.** [TanStack Query](https://tanstack.com/query)
  owns the client cache; mutations are O(1-row) REST calls with optimistic
  updates. There is no hand-rolled sync/merge layer and no IndexedDB blob.
- **Preferences** (base currency, theme) are the only non-collection state and
  live in a one-row-per-user `user_preferences` table (small opaque JSONB).

## Data model

| Collection   | Table             | Key columns |
|--------------|-------------------|-------------|
| wallets      | `wallets`         | `balance_cents`, `currency`, `sort_order`, `is_archived` |
| categories   | `categories`      | `kind`, `parent_id`, `sort_order` |
| transactions | `transactions`    | `amount_cents`, `wallet_id`, `category_id`, `to_wallet_id`, `date`, `tags` |
| budgets      | `budgets`         | `category_ids`, `period`, `limit_cents`, `alert_at` |
| goals        | `goals`           | `target_cents`, `current_cents`, `target_date` |
| recurring    | `recurring_rules` | `template`, `frequency`, `every`, `next_run_at` |
| debts        | `debts`           | `direction`, `amount_cents`, `due_date`, `is_settled` |

Every table carries `id` (text, client-generated for optimistic UI),
`user_id`, `created_at`, `updated_at`, and a nullable `deleted_at` tombstone
(soft delete → the Trash view; `… /purge` hard-deletes). All amounts are
integer **cents** (`BIGINT`); the API coerces them back to `Number` on read so
client arithmetic never silently concatenates strings.

## REST surface

Generated by a single generic factory
([`server/lib/entityRouter.js`](../server/lib/entityRouter.js)) from the
per-entity config ([`server/lib/entityConfig.js`](../server/lib/entityConfig.js)):

```
GET    /api/:entity            list (incl. soft-deleted, for Trash)
POST   /api/:entity            create (client-supplied id; ON CONFLICT idempotent)
PUT    /api/:entity/:id        update
DELETE /api/:entity/:id        soft delete
POST   /api/:entity/:id/restore
DELETE /api/:entity/:id/purge
GET/PUT /api/preferences
POST   /api/import             transactional whole-account restore (backup)
```

All routes are scoped to the authenticated user (`user_id` comes from the
session, never the request body — see `server/utils/userScope.js`). Mutations
carry an `Idempotency-Key` and are rate-limited (60 writes/min/user).

## Client

- [`src/queries/finance.js`](../src/queries/finance.js) — one query per
  collection.
- [`src/lib/finance/store.jsx`](../src/lib/finance/store.jsx) — a thin
  `FinanceProvider` facade over TanStack Query exposing
  `{ state, loaded, upsert, remove, restore, purge, setPrefs, importBackup }`.
  Pages are unchanged: they still read `state.transactions` etc. and call
  `upsert(...)`, but each call is now a single-row REST request with an
  optimistic cache update.

## What this replaced (v1)

The original app persisted the **entire** user state — every wallet,
transaction, budget, etc. — as one `states.state_json` JSONB blob, mirrored
to the server with a last-write-wins strategy and a "chunked sync" fallback
for blobs that grew past the request-size limit. That model is the reason this
rebuild exists. Its failure modes:

- **Unbounded row growth.** One user's blob is the whole account; large
  accounts hit the body-size ceiling and needed chunking just to save.
- **Whole-state writes.** Changing one transaction rewrote the entire blob —
  O(state), not O(change).
- **Lost-update races.** Two devices editing concurrently clobbered each
  other; the client grew tombstone/merge/version-counter logic to compensate.
- **Admin analytics timed out.** Counting a user's transactions meant
  `jsonb_array_length` / `jsonb_array_elements` scans over megabyte blobs.

v2 removes all of it: per-entity tables make growth bounded and indexed, writes
cost one row, concurrent edits touch different rows, and admin stats are plain
indexed `COUNT(*)` queries.

## Migration

`states` is retained only as a historical record; nothing reads it at runtime.
[`server/scripts/backfill-entities.js`](../server/scripts/backfill-entities.js)
(`npm run migrate:backfill`) copies each user's blob into the normalized tables
idempotently.
