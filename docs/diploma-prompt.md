## КОНТЕКСТ: ПОЛНОЕ ОПИСАНИЕ ПРОЕКТА «KOSHYK»

### 0. Краткая суть
**Koshyk** — персональный трекер финансов: гаманці (wallets), категории, транзакции,
бюджеты, цели (goals), повторяющиеся правила (recurring), долги (debts). Mobile-first PWA,
работает офлайн. Интерфейс двуязычный: украинский (по умолчанию) и английский.
Девиз приложения: «Особистий фінансовий трекер».

**Важнейший инженерный сюжет проекта:** изначально это был трейдерский журнал **TradeCRM**
(сделки, бэктесты, торговые идеи, публичный шеринг). Он хранил всё состояние пользователя в
**одном JSONB-блобе** (`states.state_json`) с самодельной синхронизацией last-write-wins и
«чанковой» (chunked) досылкой для больших блобов. Проект был полностью переосмыслен и
перестроен в финансовый трекер Koshyk, а затем архитектура данных была переведена с
блоб-модели (**v1**) на нормализованные таблицы по сущностям с TanStack Query (**v2**).
Диплом должен подробно раскрыть это «до/после».

### 2. Архитектура данных — v1 (антипаттерн) → v2 (нормализованная)

**v1 (TradeCRM, что заменили):** всё состояние пользователя — один JSONB-блоб
`states.state_json`, зеркалируемый на сервер по стратегии last-write-wins, с «чанковой»
синхронизацией (`/api/sync`) для блобов, превышавших лимит размера запроса, и
клиентским слоем слияния/тумбстоунов/счётчиков версий для разрешения 409-конфликтов.
Клиент хранил блоб в IndexedDB (`idbStorage`).

**Провалы v1 (это нужно подробно объяснить в дипломе как мотивацию рефакторинга):**
- **Неограниченный рост строки** — блоб одного пользователя = весь аккаунт; большие аккаунты
  упирались в лимит размера тела запроса и требовали чанкинга только чтобы сохраниться.
- **Запись всего состояния** — изменение одной транзакции переписывало весь блоб: O(state),
  а не O(change).
- **Гонки и потерянные обновления** — два устройства, редактируя одновременно, затирали друг
  друга; клиент оброс логикой слияния/тумбстоунов/версий.
- **Аналитика админки таймаутила** — подсчёт транзакций пользователя означал
  `jsonb_array_length` / `jsonb_array_elements` сканы по мегабайтным блобам.

**v2 (Koshyk, текущая модель):**
- **Сервер — единственный источник истины.** Каждая финансовая коллекция живёт в собственной
  нормализованной таблице Postgres с FK на `users` и нужными индексами. Единого «блоба
  состояния» нет.
- **Клиент — тонкий кеш поверх REST.** TanStack Query владеет клиентским кешем; мутации —
  REST-вызовы стоимостью в одну строку с оптимистичными обновлениями. Нет самодельного
  слоя синхронизации/слияния и нет IndexedDB-блоба.
- **Настройки** (базовая валюта, тема) — единственное не-коллекционное состояние, хранится в
  таблице `user_preferences` (одна строка на пользователя, маленький непрозрачный JSONB).

**Что улучшилось:** рост ограничен и индексирован; запись стоит одну строку; параллельные
правки затрагивают разные строки (нет гонок); статистика админки — обычные индексированные
запросы `COUNT(*)`.

**Миграция:** таблица `states` сохранена только как исторический артефакт (в рантайме её
никто не читает); скрипт `server/scripts/backfill-entities.js` (`npm run migrate:backfill`)
идемпотентно копирует блоб каждого пользователя в нормализованные таблицы.

### 3. Модель данных (схема БД, `server/db.js`)

Все денежные суммы — целочисленные **центы** в `BIGINT` (`amount_cents`, `balance_cents`,
`limit_cents`, `target_cents`…). API при чтении приводит их обратно к `Number`, чтобы
клиентский `+` не склеивал строки. Все финансовые таблицы имеют `id` (текстовый,
генерируемый клиентом для оптимистичного UI, с префиксом: wal/cat/tra/bud/goa/rec/deb),
`user_id` (FK), `created_at`, `updated_at` и обнуляемый `deleted_at` (мягкое удаление →
корзина; `…/purge` — жёсткое удаление).

**Финансовые таблицы:**
| Коллекция | Таблица | Ключевые поля |
|---|---|---|
| wallets | `wallets` | `balance_cents`, `currency`, `type`, `color`, `icon`, `sort_order`, `is_archived` |
| categories | `categories` | `kind` (income/expense, CHECK), `parent_id` (иерархия), `sort_order`, `is_archived` |
| transactions | `transactions` | `type` (income/expense/transfer, CHECK), `amount_cents`, `currency`, `wallet_id`, `to_wallet_id`, `category_id`, `date`, `note`, `tags`(JSONB), `recurring_id` |
| budgets | `budgets` | `category_ids`(JSONB-массив id), `period`, `limit_cents`, `alert_at`(% порог), `rollover` |
| goals | `goals` | `target_cents`, `current_cents`, `target_date`, `wallet_id` |
| recurring | `recurring_rules` | `template`(JSONB), `frequency`, `every`(N), `next_run_at`, `active` |
| debts | `debts` | `direction` (owe/owed, CHECK), `counterparty`, `amount_cents`, `due_date`, `is_settled` |

**Auth/служебные таблицы:** `users` (логин, e-mail, password_hash, роль, верификация e-mail,
смена e-mail, временные/постоянные баны `is_disabled`/`disabled_until`, поля 2FA
`totp_enabled`/`totp_secret` (шифр AES-256-GCM)/`totp_secret_pending`, кулдауны смены
имени/ника), `sessions` (sid, expires_at, revoked, ip, ua), `login_challenges` (тикеты 2FA,
10 мин), `backup_codes` (10 резервных кодов 2FA, bcrypt), `totp_used_codes` (защита от
повторного использования кода), `password_resets`, `states` (легаси-блоб, не используется),
`admin_logs` (аудит), `ip_bans`, `usage_daily` (метрики трафика), `rate_limits`,
`idempotency_keys` (кеш ответов 24ч), `user_stats_cache`, `notifications`,
`user_preferences` (JSONB ≤2KB), `exchange_rates_cache`.

**Допустимый JSONB (по правилам проекта):** только мелкие per-row метаданные, НЕ коллекции
сущностей — `user_preferences.prefs`, `transactions.tags`, `budgets.category_ids`,
`recurring_rules.template`, `notifications.data`, `admin_logs.meta_json`, тела ответов
идемпотентности.

### 4. REST API и универсальная CRUD-фабрика

API генерируется единой обобщённой фабрикой `server/lib/entityRouter.js` из конфигурации
сущностей `server/lib/entityConfig.js`. Конфиг описывает каждую сущность как набор полей
`{ key, col, kind }`, где `kind` ∈ {text, num, bool, ts, date, json}; функции:
- `rowToDto` — строка БД → DTO клиента (BIGINT→Number, timestamps→ISO, date→YYYY-MM-DD);
- `dtoValueToParam` — DTO → SQL-параметр (поддержка частичных обновлений);
- `validateEntity` — валидация обязательных полей и диапазонов сумм (≤ 1 трлн центов).

**Эндпоинты (на каждую сущность, монтируются автоматически):**
```
GET    /api/:entity            список (вкл. мягко удалённые — для Корзины)
POST   /api/:entity            создание (id от клиента; ON CONFLICT — идемпотентно)
PUT    /api/:entity/:id        обновление
PATCH  /api/:entity/:id        частичное обновление
DELETE /api/:entity/:id        мягкое удаление
POST   /api/:entity/:id/restore     восстановление из корзины
DELETE /api/:entity/:id/purge       жёсткое удаление
GET/PUT /api/preferences            настройки пользователя
POST   /api/import                  транзакционное восстановление всего аккаунта (бэкап)
```
Все маршруты привязаны к аутентифицированному пользователю: `user_id` берётся из сессии,
никогда из тела запроса (`server/utils/userScope.js`, middleware `rejectUserIdInRequest`).
Мутации несут заголовок `Idempotency-Key` и ограничены частотой (60 записей/мин/пользователь).
`POST /api/import` — единственная легитимная запись всего аккаунта (восстановление из
бэкапа), выполняется транзакционно.

### 5. Серверная часть — детали реализации

- **Порядок middleware** (`server/app.js`): helmet (CSP, в т.ч. разрешён домен FX-курсов) →
  CORS → парсер тела (лимит настраивается `STATE_BODY_LIMIT`) → резолв сессии по подписанной
  cookie `koshyk.sid` (формат `sid.sig`, HMAC-SHA256; обработка дубликатов cookie из разных
  доменных скоупов; троттлинг `last_seen_at` раз в 2 мин) → метрики → no-store заголовки для
  `/api/*` → `ensureDb` (3 ретрая с backoff 200/500/1000мс) → монтирование роутов.
- **Маршруты:** `/api/ping`, `/api/auth`, `/api/*` (finance), `/api/admin`, `/api/health`,
  `/api/notifications`, `/api/preferences`.
- **Аутентификация:** регистрация (валидация username/password/email, bcrypt cost 14, токен
  верификации e-mail 24ч), логин (поиск по username или e-mail; **timing-safe** сравнение с
  фиктивным хешем при отсутствии пользователя — защита от перечисления; проверка
  блокировки/бана; проверка верификации e-mail), сессии (sid = UUID v4, TTL 1д или 30д по
  флагу remember). **2FA/TOTP:** настройка (секрет шифруется AES-256-GCM, QR через otplib),
  включение по 6-значному коду, 10 резервных кодов (bcrypt), защита от повтора кода
  (`totp_used_codes`), вход по тикету-челленджу (10 мин). Сброс пароля, смена e-mail с
  подтверждением, кулдауны смены display_name (7 дней) и username (30 дней).
- **Middleware:** `requireAuth` (свежий пользователь из БД, блок при is_disabled/бане),
  `requireAdmin` (role==='admin'), `idempotency` (advisory-lock Postgres сериализует ретраи,
  кеш 2xx-ответов 24ч), `rateLimitDb` (in-memory sliding window: writes 60/мин/польз.,
  login 10/5мин/IP, 2FA 5/15мин/IP, register 5/час/IP, public read 120/мин/IP),
  `metrics` (байты/латентность в `usage_daily`), `ensureDb`.
- **Админ-функции:** список пользователей со статистикой, создание/изменение/удаление
  пользователей, бан/разбан (временный `disabled_until` и постоянный `is_disabled`),
  IP-баны, дашборд (сводка, метрики по диапазону дат, топ-пользователи), аудит-лог,
  бэкапы (gzip JSON в БД), обновление кеша статистики. Аналитика читает нормализованные
  таблицы (индексированные `COUNT(*)`), без сканов по JSONB.
- **Уведомления:** типы `budget_warn`, `budget_exceeded`, `recurring_due`, `goal_reached`,
  `reminder`, `system_message`; дедупликация по `dedupeKey`; пагинация; пометка
  прочитанным; «дайджест дня».
- **E-mail (Resend):** брендированные HTML-шаблоны (верификация, сброс, смена e-mail);
  валидация `APP_URL` (запрет Vercel-preview URL в проде).
- **Корректное завершение (graceful shutdown):** ожидание in-flight запросов до 10с,
  закрытие пула, hard-exit на 25с; периодическая чистка истёкших сессий/токенов/идемпотентных
  ключей раз в час.

- **CI/CD** (`.github/workflows/ci.yml`): `npm ci` → **check:patterns** (первым, <1с) →
  **lint** → **test** → **build**.
- **Структурный guard** (`scripts/check-patterns.sh`): grep-проверки, падающие при возврате
  v1-антипаттернов: вызовы удалённых `/api/state`/`/api/sync`; токены
  `syncDb`/`syncChunked`/`mergeStates`/`mergeArraysById`/`reconcile*`; `idbStorage`;
  `state_json` в routes/services; read-modify-write целой коллекции в теле запроса.
- **ESLint (flat config):** `no-restricted-imports` (ошибка) на `idbStorage`/`syncDb`/
  `syncChunked`; react-hooks правила; console — warning (кроме скриптов/тестов).
- **Правила проекта (`CLAUDE.md`)** — 5 «жёстких» правил, которые архитектура соблюдает
  структурно: (1) никаких JSONB-блобов для коллекций сущностей; (2) деньги — целочисленные
  центы; (3) сервер — источник истины, синхронизация только через TanStack Query, никакого
  самодельного sync/merge; (4) мутации стоят O(change), не O(state); (5) ошибки —
  структурированные, со стабильным `code` (NOT_FOUND, INVALID_PAYLOAD, RATE_LIMITED,
  DB_UNAVAILABLE…), без логирования секретов.
- **Документация:** `docs/ARCHITECTURE.md` (модель v2 и что заменили), `README.md`.

### 8. Хронология разработки (нарратив «до/после» для диплома)

1. **TradeCRM (предшественник)** — трейдерский журнал на блоб-модели v1; финальные «костыли»
   синхронизации (серверное авто-слияние при 409, обработка гонок чанков, коалесинг мутаций) —
   документируют, *почему* потребовался ребилд.
2. **Ребрендинг TradeCRM → Koshyk** — полная замена предметной области на личные финансы;
   новые сущности, mobile-first оболочка, форма быстрого ввода TransactionSheet, i18n (uk/en).
3. **Раунды функционала 2–6** — повторяющиеся операции, CSV-импорт, FX и нетто-капитал,
   JSON-бэкап, тосты, бюджет-алерты, PIN-замок, фильтры/теги/reorder, цели, уведомления,
   долги, скелетоны, JSDoc; рост тестов до ~81+.
4. **Редизайн «Calm Fintech v2»** — индиго-бренд, градиенты, шрифт Geist, ProgressRing,
   переработанные примитивы и лендинг.
5. **Великая миграция v1→v2** — серверный коммит (нормализованные таблицы + универсальный
   CRUD + удаление `/api/state` и `/api/sync` + backfill) и клиентский коммит (TanStack Query
   вместо IndexedDB-блоба и самодельной синхронизации; публичный API store сохранён — страницы
   не переписывались). Внешне UI не изменился, внутри — другая архитектура.
6. **Governance** — CLAUDE.md, ESLint, check-patterns.sh, CI, чтобы v2 не сполз обратно в v1.
7. **Полировка качества продукта** — онбординг-карточка, скелетоны на списках, инлайн-
   валидация, тема на весь апп при загрузке, честная сравнительная таблица на лендинге,
   починка админ-дашборда и графиков использования.

### 9. Замечания о точности
В кодовой базе остались отдельные следы предыдущей версии (например, упоминания

архитектуру v2, изложенную выше и в `docs/ARCHITECTURE.md`.
